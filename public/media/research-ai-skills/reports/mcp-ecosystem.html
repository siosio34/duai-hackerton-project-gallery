01

요약

결론부터: MCP는 조직 차원의 AI 툴 연결 계층에서 사실상 무경쟁 표준이 됐고, "클라이언트 2종 이상 × 통합 3개 이상 × 중앙 감사 요건"을 충족하는 팀이라면 지금 도입하는 것이 합리적이다. 단 게이트웨이 뒤에서, 검증된 서버만, 동시 활성 툴 수를 예산제로 관리하면서다. 신규 서버 구축이라면 세션 모델을 갈아엎는 2026-07-28 무상태 스펙[S4]이 안착한 뒤 시작하는 편이 마이그레이션 비용 한 번을 아낀다. 그 임계점에 못 미치는 팀은 CLI·직접 API 통합이 더 싸고 단순하다 — 이는 Thoughtworks Tech Radar의 비벤더 권고와도 일치한다[S63].

판단의 근거 세 가지. 첫째, OpenAI·Google·Microsoft·AWS가 모두 채택했고 2025-12 Linux Foundation으로 거버넌스가 중립화됐다[S6]. 둘째, "수만 개 서버" 생태계의 실질 가용 코어는 수천 개 단위이며(독립 감사 기준 17%만 프로덕션급[S25]) 서버 품질 검증이 도입의 실제 병목이다. 셋째, 보안 사고의 다수는 연구 데모였지만 실전 공급망 공격이 분기 1–2건 페이스로 누적 중이고[S39][S65], NSA가 전용 지침을 낼 만큼[S67] 리스크는 실재한다. 가장 큰 캐비앗: 스펙은 아직 4–6개월 주기로 변하므로, 연 1–2회의 스펙 대응 비용을 도입 비용에 포함시켜야 한다.

02

핵심 인사이트

1MCP는 사실상 업계 표준이 됐지만, 스펙은 아직 "움직이는 표적"이다

OpenAI(2025-03)[S8], Google(2025-04)[S10], Microsoft·AWS(2025-05)[S12][S13]가 모두 채택했고 2025-12 Linux Foundation으로 중립화됐다[S6]. 그러나 19개월간 5개 리비전이 나왔고, 2026-07-28 차기 스펙은 세션 모델을 갈아엎는 "출시 이후 최대 개정"이다[S4]. 표준 채택과 스펙 안정은 별개의 문제다.

2"수만 개 서버" 숫자는 믿지 말 것 — 실질 가용 코어는 수천 개 단위다

같은 시기 디렉토리별 집계가 공식 레지스트리 11,646개[S16]에서 Glama 34,010개[S20]까지 3배 벌어진다. 독립 감사에서는 고유 프로젝트의 52%가 사실상 방치 상태, 17%만이 프로덕션에 쓸 만한 수준이었다[S25]. 생태계의 크기가 아니라 개별 서버의 품질을 검증해야 한다.

3보안 사고는 "대부분 연구 데모"가 맞지만, 실전 사례가 분기 1–2건 페이스로 늘고 있다

널리 회자되는 GitHub·Supabase 사건은 통제된 연구 시연이었다[S32][S34]. 그러나 postmark-mcp 백도어(2025-09)[S39], Smithery 공급망 침해(2025-10)[S65], 인포스틸러 탑재 가짜 서버(2026-02)[S43] 등 실전 사례가 누적 중이고, NSA가 2026-05 전용 보안 지침까지 발행했다[S67].

4최대 운영 비용은 보안이 아니라 컨텍스트 잠식이다

툴 정의 하나가 550–1,400토큰을 차지해, 서버 5–10개 스택이 컨텍스트 창의 25% 이상을 입력 전에 소모하는 사례가 반복 보고된다[S55][S59]. 툴이 늘수록 모델의 툴 선택 정확도가 급락한다는 연구도 있다[S58]. 도입 시 "동시 활성 툴 수 예산제"가 필수다.

5"MCP 사망론"의 실체는 사망이 아니라 용도 분화다

로컬·개인 개발에서는 CLI와 Skills가 더 싸고 단순하다는 합의가 형성됐지만[S49][S60], 회의론 진영조차 중앙 감사·자격증명 위임이 필요한 조직 배포에서는 MCP의 유효성을 인정한다[S61][S63]. 기업 실명 이탈 사례는 Perplexity 1건(보도 기준)뿐이다[S62].

6신규 구축이라면 2026-07-28 무상태 스펙 안착 후 시작이 마이그레이션 1회분을 아낀다

차기 스펙이 프로토콜 세션을 제거하고 Roots·Sampling·Logging 3개 프리미티브를 폐기 예고(12개월 수명주기)했다[S4]. 지금 이 기능들에 의존하는 설계를 시작하는 것은 예고된 부채를 떠안는 일이다. 반대로 이미 다중 클라이언트 요구가 있다면 기다릴 이유가 없다 — 게이트웨이 생태계는 충분히 성숙했다[S57][S46].

03

상세 분석

여섯 개의 질문을 따라간다 — 표준은 단단한가(§3.1), 생태계는 실제로 얼마나 큰가(§3.2), 무엇이 실제 보안 사고이고 무엇이 과장인가(§3.3), 어떻게 방어하는가(§3.4), "MCP는 죽었다"는 담론의 실체는 무엇인가(§3.5), 그래서 지금 도입해야 하는가(§3.6).

표준은 빠르게 변하지만, 변화 관리가 제도화되고 있다

MCP는 2024-11 출시[S15] 후 19개월 동안 네 개의 안정판(2024-11-05, 2025-03-26, 2025-06-18, 2025-11-25)을 냈고, 다섯 번째인 2026-07-28 버전의 릴리스 후보가 2026-05-21 확정됐다[S1][S2][S3][S4]. 리비전 간격은 4–6개월. "아직 안정적이지 않다"는 평가의 구체적 근거가 두 가지 있다. 첫째, JSON-RPC batching이 2025-03에 추가됐다가 2025-06에 제거됐다 — 한 사이클 만의 번복이다[S1][S2]. 둘째, 차기 스펙은 initialize 핸드셰이크와 세션 ID를 제거하는 무상태(stateless) 코어 전환으로, 스튜어드 스스로 "출시 이후 최대 개정"이라 부른다[S4].

반대 방향의 신호도 분명하다. 스펙 변경은 이제 SEP(Specification Enhancement Proposal) 절차를 거치고[S5] 연간 로드맵으로 방향이 공개되며[S14], 차기 버전은 "기능 폐기 예고 후 최소 12개월 유지"라는 수명주기 정책을 도입하며, 코어는 동결하고 실험 기능은 Extensions로 분리한다[S4]. Roots·Sampling·Logging 세 프리미티브는 제거가 아니라 폐기 예고 상태다 — 대체 경로와 12개월 유예가 함께 공지됐다[S4]. 요컨대 "변화는 빠르지만, 변화의 예측 가능성은 높아지는 중"이 정확한 진단이다.

거버넌스는 2025-12-09 Anthropic이 MCP를 Linux Foundation 산하 Agentic AI Foundation(공동 설립: Anthropic·Block·OpenAI)에 기증하며 벤더 중립화됐다[S6][S28][S7]. 기술 결정권은 회사가 아닌 개인 단위의 Steering Group이 갖고, 현재 Lead Maintainer는 Anthropic과 Microsoft 소속 각 1인이다[S5]. 그 전에 이미 OpenAI(2025-03)[S8][S9], Google DeepMind(2025-04)[S10], Microsoft·GitHub·AWS(2025-05)[S11][S12][S13]가 공식 채택을 발표했다. 경쟁사 전원이 표준을 공동 운영하는 단계까지 온 프로토콜은 이 영역에서 MCP가 유일하다.

불확실 — 2026-07-28 스펙은 아직 릴리스 후보(RC) 단계다. 본 보고서의 차기 스펙 서술은 RC 기준이며, 최종판에서 세부가 달라질 수 있다[S4]. AAIF 재단의 이사회 구성·예산 등 내부 거버넌스 상세도 공개 자료로 확인되지 않았다.

생태계: "수만 개"는 등록 수, 쓸 만한 것은 수천 개

2026-06-11 본 조사에서 공식 레지스트리 API를 직접 페이지네이션 집계한 결과 최신(latest) 서버 레코드는 11,646개로, 18일 전 제3자 집계 9,652개[S23] 대비 20.7% 늘었다[S16]. 같은 시기 서드파티 디렉토리는 Glama 34,010개[S20], mcp.so 약 19,700개[S22], PulseMCP 17,766개[S21]를 표방한다 — 최대 3배 괴리다. 디렉토리 수치의 성장 곡선 자체는 가파른 것이 사실이다(예: Smithery 2024-12 10개 → 2025-04 4,533개[S30]). 원인은 수집 방식(스크레이핑 vs 등록), 레지스트리 간 중복 등재, 그리고 버전 레코드 혼동(공식 레지스트리도 버전 포함 시 28,959개)이다[S25][S23].

숫자보다 중요한 것은 품질 분포다. 2026-04 한 독립 감사는 1,847개 고유 프로젝트를 추적해 52%가 dead/방치 상태, 17%만 프로덕션 사용을 고려할 만하다고 보고했다[S25]. 1년간 MCP 인프라를 운용한 StackOne CTO의 "공개 서버 대부분은 프로덕션 품질 미달" 평가[S66], 그리고 공식 레지스트리 자체가 아직 preview 단계로 동작 검증·보안 스캔을 하지 않는다는 사실[S17]이 같은 방향을 가리킨다. 즉 "생태계가 크니 안전하다"는 추론은 성립하지 않는다.

상충 — Anthropic은 2025-12 기증 발표에서 "활성 공개 서버 10,000개 이상"이라 했고[S6], 4개월 뒤 독립 감사는 "52%가 dead"라 했다[S25]. 두 수치는 'active'의 정의 차이(등록·도달 가능 vs 유지보수됨)로 양립하지만, 도입 의사결정에는 후자의 정의가 유효하다.

반면 수요 측 신호는 강하다. 클라이언트 지원은 사실상 보편이다: ChatGPT(2025-09 풀 클라이언트화)[S27], VS Code/Copilot(2025-07 GA)[S26], Gemini CLI·Google Cloud[S29], Claude·Cursor[S6]. 서버를 한 번 구현하면 모든 주류 AI 클라이언트에 도달한다는 것 — 이것이 직접 API 통합이 가질 수 없는 MCP의 구조적 우위다. SDK 다운로드도 본 조사 실측 기준 월 4.37억 회(TypeScript 154.8M + Python 282.8M)로[S18][S19], Anthropic이 6개월 전 발표한 "월 9,700만"[S6]의 약 4.5배다. 단 패키지 다운로드는 CI/봇 트래픽을 포함하므로 사용자 수가 아닌 추세 지표로만 읽어야 한다.

불확실 — 프로덕션 채택률로 유통되는 "조직의 41–45%"는 MCP 보안 벤더 Stacklok의 n=100 단일 서베이가 유일한 출처이며 독립 재현이 없다[S24]. 참고치 이상으로 쓰지 말 것. 한때 회자된 "78% 엔터프라이즈 채택"은 무출처로 판명·철회됐다[S23].

보안: 공격 클래스는 7종, 실전 사고는 3건, 나머지 유명 사건은 연구 데모다

문서화된 공격 클래스는 일곱 가지로 정리된다. ① 툴 설명 메타데이터에 악성 지시를 숨기는 tool poisoning — 툴을 호출하지 않아도 목록 조회 시점에 주입되므로 "호출 시 승인" 모델을 우회한다[S33][S41]. ② 승인 후 툴 정의를 바꾸는 rug pull[S33]. ③ 신뢰 서버의 동작을 다른 서버가 재프로그래밍하거나, 외부 데이터(이슈·티켓)를 통해 간접 프롬프트 주입이 일어나는 toxic agent flow[S32][S34]. ④ OAuth 프록시의 confused deputy·토큰 패스스루[S35]. ⑤ 공급망 공격·typosquatting — 레지스트리 11곳 중 9곳이 무심사로 위장 패키지를 게재했다는 통제 연구가 있다[S42]. ⑥ 툴체인 자체의 Critical CVE — 악성 서버 접속만으로 클라이언트가 원격 코드 실행되는 mcp-remote CVE-2025-6514(CVSS 9.6)[S31], MCP Inspector CVE-2025-49596(9.4)[S40]. ⑦ 단순 노출 — 인터넷에 무인증 공개된 MCP 서버 1,862대(2025-07 스캔)[S37].

"실제로 일어난 일"과 "연구 시연"의 구분이 이 주제에서 가장 흔히 왜곡되는 지점이다. 실전(in-the-wild) 확정 사례는 세 건이다: 발신 메일을 통째로 빼돌린 postmark-mcp npm 백도어(2025-09, 최초의 실전 악성 MCP 서버)[S39], Smithery 레지스트리 공급망 침해(2025-10)[S65], 인포스틸러를 탑재한 가짜 Oura 서버(2026-02)[S43]. 여기에 악용 증거는 없지만 실제 발생한 Asana 교차 테넌트 노출 버그(2025-06)가 더해진다[S38]. 반면 가장 유명한 GitHub MCP "프라이빗 저장소 유출"과 Supabase "lethal trifecta"는 통제된 연구 데모였고[S32][S34], Amazon Q 와이퍼 사건은 MCP 사건이 아닌데도 MCP 사고 목록에 자주 잘못 포함된다[S44].

그렇다고 "데모뿐이니 안전"한 것은 아니다. 제3자 사고 타임라인 기준 실전 사례는 분기 1–2건 페이스로 늘고 있고[S65], 오픈소스 서버 1,899개를 스캔한 연구는 7.2%에서 일반 취약점, 5.5%에서 tool poisoning 취약점을 발견했으며 그중 상당수는 기존 SAST 도구로 잡히지 않는 MCP 고유 유형이었다[S36]. 2026-05에는 NSA가 MCP 전용 보안 설계 지침(CSI)을 발행했다 — 이 리스크가 보안 벤더의 마케팅만은 아니라는 비상업적 확인이다[S67]. 한편 "MCP의 설계 결함" 프레임(OX Security)에 대해서는 당사 프로젝트들이 "예상된 동작이며 임베딩 애플리케이션의 입력 검증 문제"라고 공식 반박했다[S64][S42] — 본 조사의 판단도 리스크의 본질은 프로토콜 결함보다 (a) 에이전트 아키텍처의 프롬프트 주입 본질 문제, (b) 미성숙한 구현·배포 관행, (c) 레지스트리 무심사에 있다는 쪽이다.

방어: 스펙은 빠르게 단단해졌고, 조직 배포의 표준 패턴은 게이트웨이다

인증 스펙의 진화가 가장 극적이다. 2025-06-18 스펙은 MCP 서버를 OAuth 2.1 리소스 서버로 규정하고, RFC 8707로 토큰의 수신자(audience) 검증을 의무화했으며, 토큰 패스스루를 명시적으로 금지(MUST NOT)했다[S35][S45]. 그러나 2025-08 실무 진단은 "기업은 익명 동적 클라이언트 등록(DCR)을 허용하지 않는다"며 엔터프라이즈 비현실성을 비판했고[S47], 표준 측은 2025-11-25 스펙에서 DCR을 CIMD(Client ID Metadata Documents)로 대체하고 기업 IdP가 토큰 교환·중앙 회수를 통제하는 Enterprise-Managed Authorization을 도입하며 응답했다[S3][S56]. 비판과 수정 사이의 시차는 3개월이었다 — 표준의 대응 속도 자체는 긍정 신호다.

불확실 — 새 인가 모델(CIMD·Enterprise-Managed Authorization)을 Okta·Entra 등 주요 IdP가 실제로 구현했는지는 본 조사에서 확인하지 못했다. 2026 상반기 현재 많은 팀이 여전히 API 키·게이트웨이 브로커링으로 운용 중이라는 실무 신호가 있다[S53].

배포 패턴은 "개발은 로컬 stdio, 프로덕션은 원격 Streamable HTTP + 게이트웨이"로 수렴했다. Cloudflare는 로컬 배포를 배제한 엔터프라이즈 레퍼런스 아키텍처(중앙 모노레포, 기본 거부 쓰기 권한, 감사 로깅)를 공개했고[S46], Kong은 Gateway 3.12(2025-10)에 MCP 프록시와 OAuth 플러그인을 탑재했다[S57]. 게이트웨이가 맡는 일은 자격증명 브로커링(사용자 토큰 비노출), 기능 단위 접근제어, 양방향 트래픽 스캐닝, 감사 로깅이다[S53]. 단, 이 "게이트웨이 필수론"의 출처가 전원 게이트웨이 판매사라는 점은 감안해야 한다 — 다만 NSA 지침[S67]과 Thoughtworks의 거버넌스 가치 인정[S63]이 방향성을 독립적으로 보강하므로, "필수"가 아닌 "조직 배포의 사실상 표준 패턴"으로 읽는 것이 정확하다. 프로토콜 차원에서도 상태풀 세션이 게이트웨이 라우팅과 오토스케일링을 방해한다는 점을 스튜어드가 인정하고[S51], 2026-07-28 스펙에서 세션을 제거한다[S4].

운영 수칙은 실제 사고에서 직접 도출된다: 저장소 단위로 스코프를 좁힌 토큰(GitHub 데모 교훈)[S32], read-only 기본값과 만능 자격증명 금지(Supabase 데모 교훈)[S34], 버전 핀닝·출처 검증(postmark-mcp 교훈)[S39], 신뢰 서버만 연결(mcp-remote RCE 교훈)[S31], 내부 서버의 인터넷 직접 노출 금지(1,862대 노출 교훈)[S37], 멀티테넌트 기능의 격리 테스트(Asana 교훈)[S38]. 스캐너로는 tool poisoning 정적 탐지와 런타임 감사를 겸하는 mcp-scan[S33], 툴 설명 변경을 탐지하는 mcp-context-protector[S41]가 있다.

"MCP는 죽었다" 담론: 로컬 개인 용도의 후퇴, 조직 거버넌스 용도의 존속

2026년 상반기 가장 큰 서사 전환은 "CLI와 Skills가 MCP를 대체한다"는 담론이다. 고참여 HN 스레드는 컨텍스트 비대, "SOAP의 재발명", 사후 땜질 OAuth를 비판했고[S49], "MCP is dead" 류의 글이 반복적으로 화제가 됐다[S60][S50]. 실측 근거도 있다: 툴 정의가 컨텍스트 창의 25–30%를 잠식하는 실배포 사례[S55], 툴 수 증가 시 선택 정확도가 43%에서 13.6%로 붕괴한다는 preprint[S58]. Anthropic 스스로 코드 실행 패턴으로 토큰 소비를 98.7% 줄였다고 발표했는데[S48], 이 글이 "Anthropic도 MCP 비효율을 자인했다"는 논거로 역인용되고 있다 — 원문 논지는 대체가 아니라 효율화다.

그러나 이탈의 실증은 얇다. 실명 이탈 사례는 Perplexity의 사내 MCP 후퇴 발언 1건이고, 그마저 1차 트랜스크립트 없이 2차 보도로만 확인된다[S62]. "Google이 MCP를 버렸다"는 루머는 반증됐다 — Gemini CLI의 교체는 플랫폼 전환이며 Workspace·Cloud의 MCP 지원은 지속된다[S68]. OpenAI Apps SDK의 분기 우려도 대체로 기각된다 — Apps SDK는 2025-11 SEP로 MCP Apps 표준에 병합됐고 OpenAI 문서가 표준 우선을 권고한다. 다만 커머스 등 ChatGPT 전용 확장은 부분 락인으로 남는다[S69].

가장 무게 있는 비벤더 평가는 Thoughtworks Tech Radar다: 2025-11 "단순 API→MCP 변환"을 Hold로, 2026-04에는 "기본값으로서의 MCP" 자체에 주의를 권고하며 "잘 설계된 CLI + 구조화된 출력으로 충분한지 먼저 평가하라"고 했다[S63]. 주목할 점은 회의론 진영조차 중앙 감사·자격증명 위임·다중 클라이언트라는 조직 요건에서는 MCP의 유효성을 인정한다는 것이다[S60][S61]. 종합하면 이것은 사망이 아니라 층위 분화다: 개인·로컬 에이전트의 기본값은 CLI/Skills로 이동했고, 조직의 거버넌스 계층으로서의 MCP는 게이트웨이·표준 투자가 오히려 확대되고 있다[S57][S4].

도입 타이밍: 요건이 맞으면 지금, 신규 구축이면 7월 이후

지금 도입이 합리적인 조건은 세 가지다. ① 동일한 내부 툴을 다중 클라이언트(Claude·IDE·CI·사내 에이전트)에 노출해야 할 때 — 이것이 MCP의 구조적 본령이다[S49][S54]. ② 중앙 감사·자격증명 브로커링이 컴플라이언스 요건일 때 — 게이트웨이 생태계가 2025-10 이후 충분히 성숙했다[S57][S46]. ③ 통합 대상이 3개 이상이고 에이전트가 런타임에 툴을 선택해야 할 때[S54].

반대로 단일 모델·소수 고정 엔드포인트라면 직접 API나 CLI가 더 싸고 단순하며[S54][S63], 신규 서버 구축은 2026-07-28 무상태 스펙 안착 후 시작하는 편이 마이그레이션 1회분을 절약한다. 특히 폐기 예고된 Roots·Sampling·Logging에 의존하는 설계는 지금 시작하지 말아야 한다[S4]. 도입하는 경우의 가드레일: 게이트웨이 뒤에만 배포하고 기능 단위 allowlist를 적용[S53], 동시 활성 툴 수를 예산제로 관리(클라이언트별 상한 80–128 보고)[S55]하고 lazy loading·Tool Search를 활성화[S59], 서버·IdP 선정 시 2025-11-25 인가 스펙 지원 여부를 기준에 포함[S56], 그리고 연 1–2회 스펙 대응 비용을 로드맵에 명시[S4].

분기별 관찰 지표: 7월 무상태 스펙의 SDK·서버 지원율[S4], CIMD·Enterprise-Managed Authorization의 IdP 구현 출시[S56], 게이트웨이 시장의 통합 움직임[S53], 그리고 Perplexity류 이탈이 추세가 되는지 일화로 남는지[S62].

04

비교 분석

먼저 층위를 구분해야 한다. OpenAI function calling은 모델이 함수를 호출하는 인터페이스, MCP는 툴을 배포·검색·전송하는 표준, A2A는 에이전트 간 통신이다 — MCP 서버의 툴도 결국 각 모델의 function calling으로 실행되고, A2A로 위임받은 에이전트가 MCP로 툴을 호출한다. A2A는 공식적으로 MCP의 경쟁이 아닌 보완 관계다[S52]. 따라서 실제 의사결정은 "통합을 어떤 계층에서 표준화할 것인가"의 문제다.

표가 말하는 것: 트레이드오프의 핵심 축은 도달 범위 vs 컨텍스트 비용이다. MCP는 "한 번 구현, 전 클라이언트 도달"을 사는 대신 토큰 비용과 스펙 추적 비용을 지불한다. 이 거래가 이득이 되는 임계점은 대략 "클라이언트 2종 이상 × 통합 3개 이상 × 중앙 감사 요건"이고, 그 아래에서는 직접 통합이나 CLI가 우월하다[S54][S63]. OpenAPI 자동 변환은 출발점으로만 쓰고, 변환 후 공격적인 툴 가지치기와 LLM 관점의 설명 재작성이 없으면 Thoughtworks가 Hold 판정한 함정에 빠진다[S63].

05

리스크와 기회

리스크

  • 공급망 공격의 실전화 영향 높음·빈도 증가 중 — postmark-mcp 백도어, Smithery 침해, 가짜 서버 등 실전 사례가 분기 1–2건 페이스[S39][S65]. 레지스트리 무심사가 구조적 원인[S42].
  • 2026-07-28 스펙 컷오버 발생 확실·영향 중간 — 세션 제거, 3개 프리미티브 폐기 예고. 기존 구현은 12개월 내 마이그레이션 계획 필요[S4].
  • 컨텍스트 잠식과 정확도 저하 발생 흔함·영향 중간 — 무관리 툴 스택이 컨텍스트 25%+를 소모하고 툴 선택 정확도를 떨어뜨림[S55][S58].
  • 커뮤니티 서버 품질 편차 — 독립 감사 기준 17%만 프로덕션급[S25]. 서버별 검증 없는 도입은 보안·신뢰성 양쪽의 부채.
  • 인가 모델 과도기 — 새 엔터프라이즈 인가 스펙의 IdP 구현 보급이 미확인 상태. 그 공백을 API 키 관행이 메우는 중[S56][S53].

기회

  • "한 번 구현, 전 클라이언트 도달" — ChatGPT·Gemini·Copilot·Claude가 전부 MCP 클라이언트. 내부 툴 표준화의 배포 채널로 대체재 없음[S27][S26].
  • 중립 거버넌스 안착 — Linux Foundation 이관 + 경쟁사 공동 운영으로 단일 벤더 락인 리스크가 구조적으로 해소[S6][S28].
  • 무상태 전환으로 운영 단순화 — 7월 스펙이 게이트웨이 라우팅·오토스케일링 마찰을 제거. 늦게 진입하는 팀일수록 과거 부채 없이 시작[S4][S51].
  • 거버넌스 도구 성숙 — 게이트웨이(Kong·Cloudflare 등)·스캐너(mcp-scan)·공식 보안 가이드가 갖춰져 "안전하게 도입하는 법"이 문서화됨[S57][S33][S35].
  • 에이전트 스택의 디폴트 포지션 — A2A·Skills와의 역할 분담이 정리되며, 조직-툴 연결 계층의 표준 자리는 사실상 무경쟁[S52].
06

결론과 전망

여섯 개 핵심 질문에 대한 직답:

  • 표준은 성숙했나? 거버넌스는 성숙(LF 중립화·SEP 절차·수명주기 정책), 스펙은 아직 유동(7월 대개정 임박). "채택해도 되는가"와 "지금 깊이 의존해도 되는가"를 분리해서 답해야 한다[S4][S5].
  • 생태계는 실재하나? 수요 측(클라이언트 지원)은 보편적으로 실재, 공급 측(서버 품질)은 헤드라인 숫자의 1/5 수준으로 할인해서 볼 것[S16][S25].
  • 보안은 위험한가? 공격 클래스는 실재하고 실전 사례는 증가 중이지만, 게이트웨이·최소권한·버전 핀닝·스캐너라는 검증된 완화 스택이 있다. 위험한 것은 MCP 자체보다 "무검증 커뮤니티 서버를 무인증으로 연결하는 관행"이다[S39][S37][S67].
  • 대안과의 선택은? "클라이언트 2종+ × 통합 3개+ × 중앙 감사 요건"이면 MCP, 그 아래면 직접 API/CLI. 기본값으로서의 MCP는 Thoughtworks와 함께 경계할 것[S63][S54].
  • 지금 도입하나? 위 요건이 맞으면 지금 — 단 게이트웨이 뒤에서, 검증된 서버만, 툴 수 예산제로. 신규 서버 구축이면 7월 무상태 스펙 안착 후 시작[S4].

결론이 바뀌는 조건: ① 7월 최종 스펙이 RC와 크게 달라지거나 안착이 지연되면 "신규 구축 대기" 권고가 연장된다. ② Perplexity류 이탈이 2026 하반기에 복수의 실명 사례로 확산되면 조직 계층에서의 MCP 우위 판단을 재검토해야 한다. ③ CIMD·Enterprise-Managed Authorization을 주요 IdP가 출시하면 게이트웨이 의존도가 낮아져 도입 문턱이 한 단계 내려간다[S56].

개발팀 리드의 다음 행동: (1) 자기 조직이 임계점(클라이언트 2종+/통합 3개+/감사 요건)을 넘는지 30분 안에 판정할 수 있다 — 넘지 않으면 CLI/직접 통합으로 종결. (2) 넘는다면 파일럿은 read-only 서버 1–2개 + 게이트웨이로 시작하고, 위 분기별 관찰 지표 4개를 분기 리뷰에 넣어라. (3) 어느 쪽이든 Roots·Sampling·Logging 의존 설계와 무검증 커뮤니티 서버 직결만은 피하라 — 이 둘이 현시점에서 가장 확실하게 예고된 부채다[S4][S25].
