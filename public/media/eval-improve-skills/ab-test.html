1. 한 줄 요약

냉장고 내부를 촬영하면 보유 재료를 인식해, 지금 만들 수 있는 레시피와 부족한 재료를 제안하는 모바일 앱. 핵심 가치는 "장보기 낭비 감소 + 메뉴 결정 피로 해소".

2. 문제 정의와 근거

  • 가설 1: 가구는 구매한 식재료의 상당 부분을 사용 전 폐기한다(음식물 쓰레기). → 출시 전 정량 근거 필요: 환경부 음식물쓰레기 통계, 자체 사전 설문(n≥100).
  • 가설 2: "뭘 해먹을지 결정"이 요리 착수를 막는 주요 마찰이다. → 타깃 인터뷰 10건으로 검증 후 진행.
  • 위 두 가설이 검증되지 않으면 기획을 보류한다(근거 없는 확신을 전제하지 않음).

3. 핵심 기능 (우선순위)

  1. 재료 인식(P0): 촬영 → 후보 재료 리스트 제시 → 사용자가 확인·수정(인식은 보조, 최종 확정은 사용자). 오인식 가능성을 UX 전제로 설계.
  2. 레시피 추천(P0): 확정 재료 기반, 부족 재료 1~2개까지 허용하는 "거의 가능" 레시피 포함.
  3. 부족 재료 구매 연결(P1): 제휴 커머스 딥링크. 수익원이자 편의 기능.
  4. 유통기한 관리(P2): 수동 입력 기반으로 시작, 인식 고도화는 후순위.

4. 기술 접근과 한계

  • 재료 인식: 범용 비전 모델로 PoC → 정확도 측정(혼동 매트릭스). 한국 식재료·포장 제품은 오인식 위험이 크므로 사용자 보정 루프를 1차 방어선으로 둔다. 목표 정확도와 측정 방법을 스프린트 0에서 정의.
  • 레시피 DB: 무단 크롤링은 저작권 리스크 → 제휴 레시피 제공처 확보 또는 자체 구축. 라이선스 검토 선행.
  • 개발: 외주/채용 결정은 비용·유지보수 관점에서 별도 판단(현 단계 미확정으로 명시).

5. 수익 모델과 검증

  • 프리미엄 구독 / 커머스 수수료 / 광고. 단, 초기엔 커머스 수수료 1개에 집중해 검증한다. 구독은 무료 사용자 리텐션(주 2회 이상 사용)이 확인된 후 도입.

6. 타깃과 시장

  • 1차 타깃: 자취·1인 가구(재료 낭비·메뉴 피로가 가장 큰 세그먼트). "모두"는 타깃이 아님.

7. 성공 지표

  • 활성화: 첫 촬영→레시피 도달률, 재료 인식 수정률(낮을수록 정확).
  • 리텐션: 주간 재방문율. 비즈니스: 커머스 전환·수수료.

8. 일정(가정, 검증 후 확정)

  • M0: 문제 검증(인터뷰·설문). M1~2: 인식 PoC + 정확도 측정. M3: 핵심 플로우 베타.
  • "3개월 MVP"는 인식 정확도 검증 결과에 종속 — 검증 실패 시 일정 재산정.

9. 리스크 (회피하지 않음)

  • 인식 정확도 미달: 가장 큰 리스크. 사용자 보정 UX + 정확도 KPI로 관리, 미달 시 피벗.
  • 레시피 저작권: 제휴/자체 구축으로 회피.

2. 1차 타깃 & 페르소나 (행동 기준)

"요리하는 사람 전부"는 폐기한다.

1차 타깃 정의 [확정]: 수도권 거주, 25~34세, 1인 가구 직장인, 주 3회 이상 집밥, 식재료를 주 1회 이상 소량 구매, 식재료 폐기·메뉴고민 스트레스를 인지하는 사람.

페르소나 — "퇴근 후 김선우, 30세"

  • 맥락: 1인 가구, 평일 저녁 7~9시 귀가. 주말에 사둔 채소·계란·두부 절반이 시들거나 버려진다.
  • 트리거: 냉장고 문을 열고 "이걸로 뭘 해먹지" 5초 고민 → 결국 배달앱.
  • 현재 대안: 배달앱, 레시피 포털("메뉴명"으로 검색해야 함), 유튜브.
  • 불만(JTBD): "있는 재료로, 장보기 없이, 10분 안에 결정하고 싶다."

핵심 사용 시나리오

  1. 진입: 퇴근 후 냉장고 앞 → 앱 실행 → 칸 촬영(또는 수동 추가).
  2. 인식·확인: 인식 재료 카드 → 틀린 1~2개 탭 수정/삭제(신뢰의 핵심).
  3. 첫 가치("Aha"): "추가 구매 0개로 지금 가능" 레시피 3개를 조리 시간순 제시 → 1개 선택 → 조리 모드.
  4. 완결: 조리 완료 체크 → "오늘 ○○ 소진" 피드백.
  5. 재방문 후크: ①유통기한 임박 푸시 [가정: 알림 동의 시] ②자주 쓰는 재료 즐겨찾기 ③주간 "버린 재료 줄임" 리포트.

활성화(Activation) 정의 [확정]: 가입 후 7일 내, 사진 인식 → 레시피 1개 선택 → 조리 모드 진입까지 완료.

3. 시장 규모 (TAM/SAM/SOM) — 단일 산식으로 통일

핵심 수정: 기존 v2는 TAM/SAM/SOM에 "혼합 ARPU(무료 포함)"를 입력으로 쓰면서 민감도에서 다시 "MAU×유료전환율"을 곱해 이중계산했고, 그 결과 4만 MAU×9,000원=3.6억(=1인당 18만원/년)이라는 불가능한 값이 나왔다. 본 개정은 매출 산식을 하나로 통일하고, 혼합 ARPU는 입력이 아니라 산출값으로만 쓴다.

3.1 단일 매출 산식 [확정]

```

연 매출(run-rate) = 구독 매출 + 광고 매출

구독 매출 = (MAU) × (유료전환율) × (월 구독료 ₩3,900) × 12

광고 매출 = (무료 MAU) × (DAU/MAU 비율) × (1인 일 광고노출) × (eCPM ÷ 1,000) × 365

```

  • 여기서 유료 ARPU(과금자 1인 기준)와 혼합 ARPU(전체 MAU 1인 평균, 무료 포함)는 다른 개념이다. 혼합 ARPU는 위 산식의 결과를 MAU로 나눈 [산출]값이며, 어디에서도 입력으로 쓰지 않는다.
  • 본 절은 동시 사용자 기준 연 run-rate다. 고객 생애가치(LTV)는 코호트·유지개월 기반으로 §7.2에서 별도로 다룬다(서로 다른 개념을 분리).

② 그러나 사전 합의한 기준은 달랐다 — 스킬은 1-pass가 놓친 결함을 드러내고, 못 고칠 땐 정직하게 멈추는가. 이 기준에선 3/3 성공: 블라인드 심판이 매번 스킬본을 1위로 골랐고, 멈출 땐 가짜 통과 대신 정체·에스컬레이션·최고점 복원으로 멈췄다.

③ 한계도 명시한다. 블라인드 심판 역시 LLM이다(단일 채점). 스킬의 native 렌즈·개선기도 같은 벤더(claude) 계열이라 자기채점 편향 여지가 있다. 페르소나 렌즈는 셋 다 3점대로 낮았지만 advisory라 게이트에서 제외됨 — 합성 역할극이라 점수 게이트로 쓰면 정직성을 깎는 방향으로 최적화되기 때문(스킬의 설계 선택). external 렌즈에서 codex 호출이 라운드마다 600초 타임아웃돼 일부 라운드는 grok 단독으로 집계됐다.

④ 결론: 이 표본에서 quality-gate-loop은 "한 번 잘 쓰기"보다 더 신뢰할 결과물을 만들었고, 만들 수 없을 땐 그 사실을 숨기지 않았다. 증거가 필요한 문서일수록(증거 천장) 이득이 컸고, 텍스트가 천장인 문서에선 이득이 좁은 대신 '정직한 멈춤'의 가치가 더 두드러졌다.
